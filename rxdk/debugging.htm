Debugging

RXDK debugging runs the title on the console and connects a symbol bridge (xboxdbg-bridge) that reads the console's memory and resolves symbols from your .pdb. Start a session with F5 (or RXDK: Debug); it builds, deploys, launches under the debugger, and attaches.

Requirements:

  • The XBE must be built with imageBuild.debug: true.
  • A .pdb must sit next to the .xbe in outputDir. (For prebuilt projects, set prebuilt.pdb.)

Breakpoints

Set line breakpoints as usual. On bind they turn solid; a hollow dot means the line couldn't be mapped (often the file wasn't compiled with debug info, or the line has no code). When a breakpoint hits, execution stops and the call stack, Locals, and Globals populate for that thread.

Locals

The Locals scope shows the current frame's variables, resolved from the PDB with correct types/sizes. RXDK uses a pure-managed PDB reader (not dbghelp) so modern Zig/LLVM local records parse correctly — scalars show a value, aggregates (structs/arrays) are expandable.

Registers

The Registers scope shows the CPU registers for the stopped thread (EAX, EBX, …, EIP, EFLAGS).

Globals

The Globals scope shows your title's global variables, formatted at their absolute address and expandable like Locals. Because a title statically links the whole XDK (Direct3D, XAPI, the CRT), there are hundreds of library globals plus compile-time constant tables pulled in from headers. To keep the pane useful, RXDK ranks globals and shows the most relevant set by default.

Visibility levels

LevelShowsExample
Title (default)Your title's own mutable globals (writable .data).g_pd3dDevice, g_pVB, timing vars
Title + constantsAlso the title's compile-time constant tables (read-only .rdata).D3DTEXTUREDIRECTENCODE and other header lookup tables
AllAlso statically-linked library globals.D3D::*, CRT internals — verbose

How the split works: RXDK maps each global to its owning module (via the PDB's section contributions). Globals from a static library (object file ends in .lib) are library globals; globals from your compiland are the title's. Of those, ones in a writable section (.data/.bss) are your real mutable state, while ones in a read-only section (.rdata) are constants. The default shows only the writable, title-owned set. If a level would be empty, RXDK automatically relaxes to the next one so the pane is never blank.

File-static / anonymous-namespace globals appear without their `anonymous namespace':: qualifier for readability (the raw name is still used internally for expansion).

Changing the level

  • Command: RXDK: Cycle Globals Visibility (Command Palette during a debug session) advances Title → Title + constants → All → Title.
  • Setting: rxdk.debugger.globalsScope (title / titleAndConstants / all).
  • RXDK Settings screen: the Debugging → Globals visibility dropdown.

Changing the level refreshes the Globals pane live — no need to relaunch or step.

Watch & Evaluate

Add expressions to the Watch pane or evaluate them in the Debug Console. Expressions resolve against the current frame's symbols and read live console memory. To expand a struct's members, expand it in Locals/Globals (member-by-member expansion is supported one level deep).

Title output

Text the title prints via printf / OutputDebugStringA appears in the Xbox Title output channel. It reveals automatically when xbox.showTitleOutput is enabled. For a standalone, always-on debug-string monitor, use RXDK: Launch xbWatson.

Tips

  • Rebuild and redeploy after code changes so the on-console image matches your symbols.
  • If Locals/Globals are empty, verify the .pdb is present next to the .xbe.
  • Start with Title globals; switch to All only when you need to inspect an XDK/CRT internal.
